# Low-code Runtime Headers Contract v0.1

## Summary

Standard contract for HTTP headers and validation context used across **web-admin**, **API Gateway**, **low-code-service**, and future **core/BFF** integrations.

Goal: consistent propagation of tenant isolation, audit metadata, tracing, and soft validation context without changing public API contracts or core business logic.

Constants and helpers live in:

- Go: `github.com/freight-platform/shared-go/lowcode`
- Frontend: `apps/web-admin/constants/apiHeaders.ts`

Related docs:

- `docs/LOW_CODE_RUNTIME_INTEGRATION_POLICY_V0.1.md`
- `docs/LOW_CODE_RUNTIME_NEXT_STEPS_V0.1.md`

---

## Headers

| Header | Required | Purpose |
|--------|----------|---------|
| `X-Tenant-ID` | **Yes** | Tenant isolation for all low-code read/write |
| `X-User-ID` | Optional | Audit actor metadata |
| `X-Request-ID` | Optional | Trace / correlation metadata |
| `Authorization` | Optional | Gateway auth (Bearer JWT) |
| `X-Low-Code-Entity-Status` | Optional | Soft validation context (header fallback) |
| `X-Low-Code-Role` | Optional | Soft validation context (header fallback) |

---

## Tenant Header

**Name:** `X-Tenant-ID`  
**Constant:** `shared-go/lowcode.HeaderTenantID` / `API_HEADER_TENANT_ID`

### Rules

- Required for all low-code endpoints
- Must be a valid UUID
- **Must not** be inferred from `entity_id` or template id
- Missing tenant → `400 TENANT_REQUIRED`
- Invalid tenant UUID → `400 VALIDATION_ERROR`

### Legacy query fallback

`tenant_id` query param is accepted as fallback on some low-code reads (existing behavior). Prefer header for all new integrations.

---

## Actor Header

**Name:** `X-User-ID`  
**Constant:** `shared-go/lowcode.HeaderUserID` / `API_HEADER_USER_ID`

### Rules

- Optional UUID of acting user
- Used for **audit metadata only** (`changed_by_user_id`)
- **Not** an authorization source — RBAC remains on gateway/core services
- Invalid UUID is ignored (no error)

Gateway sets this from JWT claims when authenticated.

---

## Request ID Header

**Name:** `X-Request-ID`  
**Constant:** `shared-go/lowcode.HeaderRequestID` / `API_HEADER_REQUEST_ID`

### Rules

- Optional correlation id for tracing
- Stored in audit log when present
- Generated by frontend on each API call (`crypto.randomUUID()`)
- Gateway middleware may propagate or generate if missing

---

## Authorization

**Name:** `Authorization`  
**Constant:** `shared-go/lowcode.HeaderAuthorization` / `API_HEADER_AUTHORIZATION`

### Rules

- Optional `Bearer <JWT>` for authenticated web-admin calls
- Gateway validates token and may inject `X-User-ID` / `X-Tenant-ID`
- Low-code-service does **not** validate JWT in v0.1 — relies on gateway/BFF for auth boundary

---

## Validation Context

Soft validation for conditional required rules on custom field values.

### Body (preferred for UI)

`PUT /api/v1/low-code/custom-field-values`:

```json
{
  "validation_context": {
    "entity_status": "DRAFT",
    "role": "SHIPPER_ADMIN"
  }
}
```

### Header fallback (for server-side / BFF)

| Header | Maps to |
|--------|---------|
| `X-Low-Code-Entity-Status` | `validation_context.entity_status` |
| `X-Low-Code-Role` | `validation_context.role` |

Body fields take precedence over headers when both are set.

### Allowed keys (v0.1)

- `entity_status`
- `role`

### Trust boundary

- `validation_context` is for **soft validation** (conditional required, preview rules)
- **Not trusted** for financial decisions, payment approval, UPD creation, or core workflow authorization

## Frontend Responsibility

`apps/web-admin/composables/useApi.ts`:

- Sends `X-Tenant-ID` from tenant store (required before API call)
- Sends `X-Request-ID` per request
- Sends `Authorization` when logged in
- `useLowCodeApi.ts` passes `validation_context` in body on custom field save

Entity detail panels pass `entity_status` from loaded core entity into preview/save context.

---

## Gateway Responsibility

`services/api-gateway`:

- Proxies `/api/v1/low-code/*` to low-code-service
- CORS allows `X-Tenant-ID`, `X-Request-ID`, `Authorization`
- Auth middleware may set `X-User-ID`, `X-Tenant-ID` from JWT
- Propagates `X-Request-ID` to downstream services

Gateway does **not** transform low-code request bodies.

---

## Low-code-service Responsibility

- Reject read/write without tenant (`TENANT_REQUIRED`)
- Filter all SQL by `tenant_id`
- Extract audit context from `X-User-ID`, `X-Request-ID`
- Merge validation context from body + low-code headers
- Never infer tenant from entity identifiers

Helpers: `shared-go/lowcode` — `TenantIDFromHeader`, `ActorIDFromHeader`, `RequestIDFromHeader`

---

## Core Services Responsibility

Core TMS services (transport-order, shipment, billing-register, etc.) use their own `tenant_id` query/body conventions in v0.1.

When core/BFF calls low-code directly in future:

- Use `shared-go/lowcode` header constants
- Pass `X-Tenant-ID` explicitly
- Optionally pass `X-User-ID`, `X-Request-ID`, validation context headers
- Do **not** embed tenant in entity_id

No core service changes required in v0.1.

---

## Security Guardrails

| Rule | Enforcement |
|------|-------------|
| Tenant required | low-code-service handlers |
| No tenant from entity_id | policy + code review |
| Actor header ≠ authorization | documented; gateway RBAC |
| validation_context not for financial gates | policy |
| Cross-tenant isolation | SQL tenant filter |
